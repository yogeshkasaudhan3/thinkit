#44 - Forgot Password (Manual Admin Reset)

What & Why

Currently, customers can sign up and log in, but there is no secure password recovery process.

Implement a temporary password recovery system that does NOT require OTP or SMS integration.

The system should allow customers to request a password reset, notify the admin, let the admin verify the customer's identity manually, generate a temporary password, and force the customer to create a new password on the next login.

This feature is temporary until OTP-based password reset is implemented in the future.

--------------------------------------------------
DONE LOOKS LIKE
--------------------------------------------------

Customer Login page contains a new:

Forgot Password?

link.

Customer enters the registered mobile number.

After submitting, the customer ALWAYS sees the same message:

"If this mobile number is registered, your password reset request has been submitted. Our support team will contact you shortly."

Never reveal whether the mobile number exists or not.

--------------------------------------------------
ADMIN PANEL
--------------------------------------------------

Create a new section:

Customers
→ Password Reset Requests

Each request should contain:

• Customer Name (if available)
• Mobile Number
• Request Date & Time
• Status

Statuses:

Pending

Completed

Rejected

Admin should be able to:

✓ View Customer

✓ Generate Temporary Password

✓ Mark Request Completed

✓ Reject Request

--------------------------------------------------
TEMPORARY PASSWORD
--------------------------------------------------

Admin clicks:

Generate Temporary Password

The system automatically generates a secure random password.

Example:

THK-7M9Q-42

or

8XRQ2LPA

Admin should NOT manually enter passwords.

--------------------------------------------------
LOGIN USING TEMP PASSWORD
--------------------------------------------------

Customer logs in using the temporary password.

Immediately after login:

DO NOT allow access to:

• Home
• Categories
• Cart
• Orders
• Profile
• Any other page

Instead redirect directly to:

Create New Password

Screen.

--------------------------------------------------
CREATE NEW PASSWORD
--------------------------------------------------

Fields:

New Password

Confirm Password

Save Password

Validation:

Minimum password rules should match the current application.

Passwords must match.

--------------------------------------------------
AFTER PASSWORD CHANGE
--------------------------------------------------

Once the new password is successfully saved:

• Temporary password becomes permanently invalid.

• Password reset request status becomes Completed.

• Customer is automatically logged in.

• Customer can now use the app normally.

--------------------------------------------------
SECURITY
--------------------------------------------------

Never store plain text passwords.

Temporary passwords must also be hashed before storing.

Never expose any password through APIs.

The customer-facing response must always be identical whether the account exists or not.

--------------------------------------------------
DATABASE
--------------------------------------------------

Add only the minimum fields required.

Example:

temporaryPasswordHash

forcePasswordChange

passwordResetRequestedAt

Do not redesign the existing authentication system.

--------------------------------------------------
OUT OF SCOPE
--------------------------------------------------

Do NOT implement:

OTP

SMS

Email reset

Google authentication changes

Any third-party authentication provider
